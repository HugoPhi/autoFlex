# Fig-06 Redis 调试日志（源码下载与 `fpconv.c` 报错）

更新时间：2026-04-02
工作目录：`/home/tibless/Desktop/auto_flex/asplos22-ae/experiments/fig-06_nginx-redis-perm`

---

## 1. 问题现象
在修复镜像缓存 EOF 后，`redis` 任务进入构建阶段，但运行中仍出现错误，表现为：

- 构建流程报缺失源文件（用户侧观测到 `fpconv.c` 相关错误）。
- 部分任务目录中 `libredis/origin` 内容异常或不完整，导致后续编译找不到预期源码。

---

## 2. 关键排查动作

### 2.1 验证上游下载地址可用性（按用户要求先 `proxy`）
执行链路：

```bash
proxy
TMP=/tmp/redis-5.0.6-check-$$; mkdir -p "$TMP" && cd "$TMP"
wget https://github.com/antirez/redis/archive/5.0.6.zip -O redis-5.0.6.zip
```

结果：
- 返回 `404 Not Found`。
- 说明旧地址 `antirez/redis` 在当前链路下不可用（或者该归档地址已变更/不可访问）。

### 2.2 对比可用地址
排障过程中验证到：
- `redis/redis` 仓库路径可作为替代来源（用于同版本源码获取）。

### 2.3 与构建失败的关联
- 当源码归档下载失败或内容异常时，后续 `libredis` 目录无法准备完整源码。
- 进一步触发编译阶段文件缺失（例如 `deps/lua/src/fpconv.c`）类错误。

---

## 3. 根因归纳
本条 bug 的核心不是编译器本身，而是“上游源码获取入口过时/不可用”导致的次生构建错误：

1. 构建脚本/依赖链仍引用 `antirez/redis` 的历史下载入口。
2. 该入口在当前环境返回 404。
3. 源码未完整落地，最终在编译阶段以缺文件形式暴露（`fpconv.c` 等）。

---

## 4. 已采取的修复思路

1. 明确将 Redis 源码下载来源从旧入口迁移到可用入口（`redis/redis`）。
2. 清理旧的 redis 工作目录与缓存结果，避免旧失败态污染新任务。
3. 重新发起 `run-wayfinder-app-redis` 验证是否越过“源码缺失”阶段。

---

## 5. 操作建议（后续稳定化）

1. 在 `apps/redis/build.sh` 或对应模板入口里，显式固定可用下载源。
2. 对下载步骤增加“失败即退出”的检查，并打印最终 URL 与文件校验信息。
3. 对关键源码文件（如 `deps/lua/src/fpconv.c`）在配置后增加存在性检查，提前失败并给出明确提示。
4. 若环境依赖代理，建议在脚本中记录 `proxy` 是否启用与生效结果，降低复现成本。

---

## 6. 本条结论

- 该问题属于“上游源码入口变更/不可用”引发的构建链断裂。
- `fpconv.c` 报错是后果，不是第一根因。
- 与第一份日志中的“镜像缓存 EOF”是两条不同故障链，已独立归档。
