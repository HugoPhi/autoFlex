## 概述

Unikraft（基线）、FlexOS（无隔离）、2 个隔离间隔（以 MPK 支持的 light（共享调用栈）和 dss（受保护和 DSS）以及 EPT）的网络吞吐量（iPerf）。

## 论文式解说与图例说明

### 图表类型
柱状图，展示不同隔离配置下的网络吞吐量对比

### Y 轴 - 吞吐量
- **单位**：Gb/s （千兆比特/秒）
- **基线值**：Unikraft 原生约 3-3.5 Gb/s
- **评估标准**：隔离开销 = (基线 - 隔离配置) / 基线 × 100%

### X 轴 - 配置分组

**图例说明**：
- **Unikraft (无隔离)**：最左侧柱子，代表纯粹的 Unikraft 性能基线，无任何虚拟化或隔离
- **FlexOS None**：无隔离的 FlexOS，展示框架自身的开销
- **FlexOS MPK-Light**：使用内存保护密钥（MPK）实现的轻量级隔离，共享调用栈以降低开销
- **FlexOS MPK-DSS**：MPK + DSS（受检的直接共享段），更强的隔离但开销略大
- **FlexOS EPT**：扩展页表（Extended Page Tables）隔离，最细粒度但开销最大
- **颜色编码**：每种配置通常用不同颜色表示（需查看实际 SVG）

### 主要性能特征
1. **最小开销**（< 3%）：FlexOS MPK-Light，最接近无隔离性能
2. **适中权衡**（5-10%）：FlexOS MPK-DSS，在隔离强度与性能间的平衡
3. **最大开销**（15-25%）：EPT，细粒度隔离的成本
4. **离群表现**：某些测试中 EPT 出现极低吞吐量（见故障排除）

## 故障排除

- **问题**：EPT 测量显示性能极低，没有超过几百 Mb/s。

  **解决方案**：这通常是由于引脚定位问题引起的。由于技术原因，EPT 测量并非像 MPK 测量那样完全引脚定位，它们使用 4 个非隔离的核心。如果这些核心设置不当（与隔离核心位于不同的 NUMA 节点，在它们上面有噪音），这可能导致性能大幅下降。

- **问题**：所有测量都被阻塞在 1.9-2.0Gb/s（而它们应该达到约 3Gb/s）。

  **解决方案**：这个问题似乎即使在正确的设置中也会不时发生，我们没有很好的解释。iPerf 是一个极其敏感的基准测试，它不仅涉及两个不同的 TCP 栈（客户端加主机），还涉及非常复杂的 TCP 行为。在这种情况下，解决方案通常是使用 `reboot` 重新启动机器。
