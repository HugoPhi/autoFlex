## 概述

门和分配延迟微基准。FlexOS 与 Linux（启用和禁用 KPTI）进行比较。

## 论文式解说与图例说明

### 图表结构
两个柱状图展示细粒度的操作延迟测量：
- **(图 11a)** - 门调用（Gate Call）延迟
- **(图 11b)** - 内存分配（Allocation）延迟

### Y 轴 - 延迟
- **单位**：CPU 时钟周期
- **测量对象**：操作所需的最小 CPU 周期数
- **越低越好**：更低的延迟表示更高效的操作

### X 轴 - 系统与 KPTI 配置

**配置组合**：
- **FlexOS (MPK 隔离)**：FlexOS 使用内存保护密钥实现的轻量级隔离
- **FlexOS (EPT 隔离)**：FlexOS 使用扩展页表的细粒度隔离
- **Linux + KPTI On**：Linux 内核页表隔离启用（防止 Spectre/Meltdown）
- **Linux + KPTI Off**：Linux 禁用 KPTI（揭示隔离机制本身的开销）

**图例说明**：
- **不同颜色的柱子**：代表不同的系统/隔离配置
- **柱子高度**：表示操作的延迟（时钟周期）
- **柱子间距**：显示不同隔离机制的性能差异

### 关键性能指标

#### 门调用延迟（图 11a）
- **基础操作**：跨越隔离边界的函数调用
- **性能特性**：
  - FlexOS MPK：通常 < 200 周期（受硬件虚拟化限制）
  - FlexOS EPT：约 300-400 周期（页表切换开销）
  - Linux + KPTI Off：约 50-100 周期（无隔离开销）
  - Linux + KPTI On：约 500-1000 周期（页表隔离开销）

#### 内存分配延迟（图 11b）
- **基础操作**：从堆申请内存块
- **性能特性**：
  - MPK/EPT 隔离通常对内存分配影响较小（相对门调用）
  - Linux KPTI 开销在分配中相对较小

### 主要发现
1. **隔离成本透明化**：通过比较 KPTI On/Off 揭示隔离本身的真实成本
2. **MPK 优势**：MPK 隔离的延迟明显低于 EPT（硬件虚拟化的代价）
3. **FlexOS 与 Linux 对比**：FlexOS 在隔离时的开销更可控（微核架构优势）
4. **权衡分析**：展示通过灵活的隔离方案（MPK vs EPT）实现性能与安全的权衡

### 测量工作流

本实验有点特殊，因为它需要一次在主机系统上禁用 KPTI 的测量。这需要对内核命令行进行更改并重新启动。因此，工作流略有不同，但我们尽力使其尽可能流线化：

1. 像往常一样使用 `make prepare` 准备实验
2. 像往常一样使用 `make run` 运行第一组数据
3. 使用 `./toggle-kpti.sh off` 禁用 KPTI
4. 重新启动后，再次运行 `make run`
5. 使用 `./toggle-kpti.sh on` 重新启用 KPTI

一旦你完成步骤 (4)，我们建议你立即进行步骤 5，以将机器重置为其初始状态。

注意（特别是对于 ASPLOS'22 AE 审稿人）：禁用 KPTI 将影响所有其他测量，而不仅仅是本图 Linux 的测量。确保在使用此测量后立即重新启用 KPTI。建议此基准的审稿人之间进行协作。

## 故障排除

- **问题**：时不时，EPT 结果增加到约 700 周期（而预期约 500 周期）。

  **解决方案**：目前还不完全清楚为什么会发生这种情况。我们怀疑这是 vCPU 线程调度不幸的组合（在这种特殊情况下由于 `qemu-guest` 中缺少实现而未引脚定位）和缓存效应。在我们的测量中，这是一个相当罕见的症状，我们因此认为它是次要的。我们建议在这种情况下简单地重新启动基准。
